In 2010, four engineers established Tangled to make and sell consumer apps during their free time. In late 2010, I became Tangled's user experience advisor, evaluating the app prototypes. The founders experienced a lot of challenges while building and finding an audience for Tangled's first two Android apps. Then, they stumbled upon a promising opportunity.

Family, friends, and colleagues consistently asked if Tangled built apps for other people. The common theme was that each person knew someone who had an app idea but did not know how to code. So in 2012, two of the founders decided to pursue the opportunity full time. Tangled pivoted its business model and began designing, building, and launching apps for other entrepreneurs.

The two founders began building the Person2Person app development platform and acquiring customers for a pilot program.

  • In 2012, Tangled acquired the first pilot customer, EPOS, and built the EPOS Android and iOS apps.
  • Then in 2013, Tangled acquired the second pilot customer, iDance, and began building the iDance iOS app.

  • In 2014, we acquired the third pilot customer, The Spot, but the customer was not ready to build until 2015.
  • In late 2014, we also acquired the fourth pilot customer, Decnut, and built both an award winning prototype and market ready app for beta testing.
  • In 2015, we acquired the fifth pilot customer, The Scene, and began building The Scene iOS iPad app.
  • By Q1 2016, we launched the minimum viable products for the Person2Person platform and all 5 pilot customer apps.

Tangled Person2Person

Overview

Tangled Person2Person is an app development platform that reuses pre-built modules to design, build, and launch custom iOS and Android apps for customers.

The key to Tangled's business model is creating apps in a scalable and repeatable way. So I collaborated with the technical architect to implement a modular platform. We agreed to a few standards, namely that the platform parts should be thoughtful, intuitive, consistent, and manageable. Additionally, we defined the process of building customer apps using the module platform. We agreed to extend our Agile Scrum sprint cycle with an Agile UX sprint 0, during which we could use Lean UX methods to plan and design how to use the platform for a customer's app.

Then, I architected a two part modular design system: a modular user interface library and a customizable theme framework. The user interface library contained standard user interface elements that combined into reusable components, which were used to build layouts. And, the theme framework used variables to globally apply customer branding consistently throughout all user interfaces.

The design system was so successful that when we acquired the fifth pilot customer, the time to design and build user interfaces was reduced by 80%. My role transitioned to an advisor. And, the iOS developer used the well documented modular design system to quickly implement the app design.

Status: Private Beta Duration: September 2013 to Present
A Modular User Interface Library

Our main goal was to build the interface from the ground up using fundamental building blocks that can be maintained in one location and reused throughout any app.

So I led the collaborative design process by leveraging my expertise in requirements analysis, cognitive psychology, and human behavior to make informed decisions about the information architecture, information design, interface design, and navigation design.

Then, I built the modular user interface components in Adobe Illustrator for high fidelity storyboards, with HTML/CSS for rapid prototypes and web apps, and in XCode for native iOS apps. We planned to build the components in Android Studio when customers were ready to launch native Android apps.

The user interface library currently contains:

  • 73 Standard and Custom Elements
  • 37 Components
  • 31 Layouts
  • Screens
  • Storyboards
Designing the User Interface Library
  1. I collaborated with the technical architect to perform a top down and bottom up analysis of functional and content requirements. During the top down analysis, we identified user interface requirements based on the business strategy and technical architecture. And during the bottom up analysis, we identified the user interface requirements based on universale design principles, iOS and Android human interface guidelines, and common user needs and expectations.
  2. I distilled the comprehensive functional and content requirements into 73 standard and custom elements, which were the fundamental building blocks for most user interfaces.
      3. Content Display Examples
    • Heading Label, Title Label, Details Label, Metadata Label, View, Avatar, Image, etc.
      • User Input Examples
    • Button, Icon Button, Hyperlink Button, Text Field, Text View, Check Box, Radio Button, Slider, Stepper, Switch/Toggle, Picker, Spinner, etc.
      • UI Bar Examples
    • Navigation Bar, Navigation Bar Title, Navigation Bar Buttons, Segmented Control Bar, Segmented Control, Tab Bar, Tab Bar Item Icon, Tab Bar Item Label, Tool Bar, Tool Bar Button Label, Tool Bar Icon Button, Side Menu, Side Menu Button Label, etc.
  3. I assembled the user interface elements into components that structured and presented content to maximize user understanding. I defined the size, position, spacing, and relationship between elements in the components to create an information hierarchy that guided user action. The following are some principles that guided my design process:
      4. White Space
    • Is there enough white space between and within groups of elements so users do not accidentally trigger an element?
      • Iconic Representation, Picture Superiority Effect
    • Are images and graphics used as much as possible to capture and retain users' attention?
      • Signal-to-Noise-Ratio
    • Are only meaningful elements and information displayed?
      • Propositional Density
    • Are important/meaningful elements larger than unimportant elements?
      • Proximity
    • Are related elements arranged near each other?
      • Progressive Disclosure
    • Are complex or low priority elements hidden until necessary or requested by the user?
      • Chunking and Serial Position Effects
    • Do groups contain less than 6 values, and are the top and bottom values the most important values for the users' decisions?
  4. I combined components into layouts, screens, and storyboards that enabled users to efficiently and effectively complete tasks.
  5. The iOS and Android developers reviewed the user interface library and we all agreed to use standardized names for library elements. Elements were named according to their type and screen position. Components were named according to the types and positions of their child elements. And, layouts and screens were named according to their primary content and function.
  6. We also agreed to extend the library for high priority content and functionality, which we defined as both integral to Tangled's business model and requested by a reliable customer.

At the end of the design process, we had a centrally located, extendable user interface library with standardized names and usage guidelines.

A Customizable Theme Framework

Our main goal was to globally apply customer branding consistently throughout an app's user interfaces without changing the underlying platform code.

So I led the collaborative design process by leveraging my expertise in cognitive psychology and human behavior to make informed decisions about the visual design.

Then, I designed and built a customizable theme framework in Xcode for native iOS apps. We planned to build the framework in Android Studio when customers were ready to launch native Android apps.

The theme framework currently supports:

  • 2 Fonts and 2 Icon Fonts
  • 9 Font Sizes
  • 2 Color Schemes with 12 Color Options
  • 139 Text Variables
Designing the Theme Framework
  1. I collaborated with the technical architect to identify all the customizable user interface elements and their style properties. We decided that customers can select which font families, colors, and icons are used throughout the app.
  2. I designed a framework using 2 fonts, 9 text sizes, and 12 colors to ensure all app content communicates a visual information hierarchy that helps users quickly process information and complete tasks. The following are some principles that guided my design process:
      3. Color
    • Are colors effectively used to call attention, group interface elements, convey meaning, or enhance aesthetics?
      • Legibility
    • Are fonts, font sizes, font colors, leading, kerning, alignment, indentation, line height, line spacing, and paragraph before/after spacing consistently used?
      • Desire Line, Wayfinding
    • Are users visually guided through the desired path of actions?
      • Entry Point
    • Is there an element that serves as the attentional entry point into the screen?
  3. I defined the standardized variable names and usage guidelines for fonts, font sizes, and colors. Font variables were named according to their usage. Font size variables were named according to their size. And, color variables were named according to their usage.
      4. Example Font Variables
    • TWCTextHeadingFont should be used to style all heading labels and title labels.
    • TWCTextBodyFont should be used to style all details labels, metadata labels, form field labels, form field values, etc.
    • TWCIconFont should be used to display scalable icons instead of image files (i.e. for tab bar items, navigation bar buttons).
      • Example Font Size Variables
    • TWCFontSizeXXL should be used to set the size of all icon fonts unless otherwise noted.
    • TWCFontSizeL should be used to set the size of Title labels, Value labels, and Button labels.
    • TWCFontSizeS should be used to set the size of Heading labels and detail labels.
    • TWCFontSizeXS should be used to set the size of Metadata labels.
      • Example Color Variables
    • TWCColor1 should be used to set the color of the elements that trigger an action (e.g. button backgrounds, checkmarks, navigation bar icons, action icons, and profile card icons).
    • TWCColor2 should be used to set the color of navigation related elements (e.g. navigation bar back button, navigation bar menu icon, tab bar icons and labels, toolbar icons and buttons).
    • TWCColor3 should be used to set the color of the Side Menu Background, Modal Screen Background, and On-Boarding Screens Background.
    • TWCColor4 should be used to set the color of the elements used to alert users (e.g. button or icon).
  4. The iOS and Android developers reviewed the theme framework and we all agreed to adhere to the usage guidelines and implement the naming standards throughout the design and development process.

At the end of the design process, we had a customizable theme framework with usage guidelines and standardized variable names that allowed us to globally apply customer branding without changing the underlying platform code for each app.

Lean UX Process

Tangled implemented an Agile Scrum sprint cycle with an Agile UX sprint 0. During sprint 0, we used Lean UX methods to plan and design how to use the platform for a customer's app.

  1. The customer identified the user problems, defined a hypothesis, and made assumptions to define content and functional requirements.
  2. Tangled's product owner gathered and organized requirements into high level user stories (epics) and detailed, basic user stories.
  3. I facilitated a workshop with the customer to explore the users, their goals, and their as-is workflows. Then, we reviewed the user stories, and I sketched the agreed upon future workflows and low fidelity storyboards for each user role. The future workflows were used to gather feedback about the order of user tasks. And, the low fidelity storyboards were used to gather feedback about the information architecture including layouts and navigation.
  4. I translated the workflows and user stories into high fidelity storyboards for the minimum viable product (MVP). The user interfaces were designed to reuse as much as possible from Tangled's modular user interface library. And, the app was styled using Tangled's customizable theme framework. The following are some principles that guided my design process:
      5. Ockham's Razor, Satisficing
    • Is the design the simplest way for users' to achieve their goal?
      • Principle of Least Resistance
    • Are tasks easy to complete?
      • Entry Point
    • Is there an element that serves as the attentional entry point into the screen?
      • Desire Line, Wayfinding
    • Are users visually guided through the desired path of actions?
      • Framing, Hick's Law
    • Are options limited and arranged so that users make a decision, especially the decision you want?
      • Forgiveness
    • Are errors easy to avoid and are the negative consequences of errors minimized?
      • Storytelling
    • Does the app communicate a consistent, meaningful story/value proposition for users?
  5. I facilitated a storyboard review workshop with the Tangled product manager, Tangled developers, and the customer. The high fidelity storyboards were used to gather feedback about the information design, visual design, and clarity of the calls-to-action.
XCode Front-End Development Process

Once we finished planning and designing in sprint 0, the iOS developer and I collaborated to build the customer's app in Xcode.

  1. While the iOS developer built the app, and I configured the user interface in Xcode using our modular user interface library and customizable theme framework. Throughout the process, I collaborated with iOS developer to unit test the app and ensure the front-end worked as expected.
  2. I tested the end-to-end app experience for each user role and worked with the developer to fix any issues.
  3. Tangled released the app to the customer for user acceptance testing before releasing to the market.

Children's Books

The first children's book series follows Mira and Miro as they make new friends, explore different environments, and learn how to be better global citizens.

I wrote and illustrated fictional, fact-based books about the relationship between humans and our environment. The narratives focus on different biomes and the ecosystem provisioning services that provide tangible materials and energy, which humans need to survive.

The series will include seven books, of which, three are completed and four have story outlines. The first book uses a space race to introduce readers to different people and their ideas. The second book uses modes of travel to introduce readers to Earth's oceans, continents, and major biomes. And, the third through seventh books use human needs to introduce readers to fourteen terrestrial biomes that provide water, food, raw materials, fuel, and medicine.

Loka Books uses fictional entertainment as a channel for educating people. So I used information from the Loka Atlas and Gazetteer to create realistic settings and plots. I researched environmental facts for each biome and identified which categories of materials and energy are derived from each terrestrial biome.
